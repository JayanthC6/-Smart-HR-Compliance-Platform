This guide is designed to help a fresher MCA graduate confidently explain and defend the **Smart HR Compliance Platform** project in technical and system design interviews at product-based companies.

## 1. Project Summary (Elevator Pitch)

"I developed a multi-tenant SaaS HR Compliance Platform using Spring Boot 3.5, Spring Security, React, and Redis, designed to automate policy management, employee onboarding, and SOC2/ISO compliance auditing. The system enforces strict tenant isolation using dynamic Hibernate filters driven by an AOP Aspect based on JWT claims, ensuring zero cross-tenant data leakage. ## 2. Complete Feature List with Technical Explanation

### 1. Multi-Tenant Architecture with Hibernate Filters
* **What it does:** Allows multiple companies (tenants) to share the same PostgreSQL database and tables safely without seeing each other's data.
* **How it works technically:** Every tenant-scoped entity extends `BaseEntity` (which contains `company_id`). We define a Hibernate `@FilterDef("tenantFilter")` with a `companyId` parameter. We created a Spring AOP Aspect (`TenantFilterAspect.java`) that intercepts all Spring Data repository calls. It checks `TenantContext` (which stores the current company UUID in a `ThreadLocal` set by `JwtAuthFilter`), unwraps the Hibernate `Session`, and executes `session.enableFilter("tenantFilter").setParameter("companyId", tenantId)`.
* **In an Interview:** Say: "We chose a shared-schema, single-database design for maximum resource efficiency. To prevent data leaks, we implemented dynamic SQL modification using Hibernate filters driven by a thread-safe ThreadLocal context and AOP, meaning developer queries never have to manually append `WHERE company_id = ...`."
* **Follow-up Questions:** *How does ThreadLocal behave with thread pools (e.g. Tomcat request threads)?* Answer: "It can leak thread-state if not cleared. We mitigated this by clearing the `TenantContext` inside a `finally` block in our `JwtAuthFilter`."

### 2. JWT Authentication with Redis Blacklist Logout
* **What it does:** Secures API endpoints and allows users to sign in stateless while verifying their identities and tenant scope. It also revokes tokens upon logging out.
* **How it works technically:** During login, `AuthService` generates a JWT containing the user ID, company ID, and role. When logging out, the client sends the active JWT to `/api/auth/logout`. The backend saves this token to Redis with a TTL equal to the token's remaining validity. The `JwtAuthFilter` intercepts every incoming API request and queries Redis to see if the token is blacklisted; if it is, the request is immediately rejected with a `401 Unauthorized`.
* **In an Interview:** Say: "Instead of database-backed stateful sessions, we used stateless JWT authentication. To solve the 'stateless token revocation' problem upon logout, we integrated a high-performance Redis cache to blacklist active tokens until their original expiration time."
* **Follow-up Questions:** *Why not store blacklisted tokens in PostgreSQL?* Answer: "PostgreSQL would require a disk read on every single API request, bottlenecking performance. Redis operates completely in-memory, keeping check times under 1–2 milliseconds."

### 3. Invite-Based Employee Registration with Token Expiry
* **What it does:** Permits HR/Admins to securely invite new employees to join their tenant company without manually creating login credentials.
* **How it works technically:** HR initiates an invite request. The backend creates a `CompanyInvite` entity, generates a secure random UUID token, sets `expiresAt` to 48 hours in the future, and saves it. The frontend creates a unique registration URL (`/register-employee?token=...`). The public registration route reads this token, validates that it exists and is not expired, extracts the `company_id` to ensure the user is registered under the correct tenant, saves the new employee, and marks the invite token as used.
* **In an Interview:** Say: "We secure employee self-onboarding by requiring cryptographically random invite tokens generated by company admins. These tokens are bound to a strict 48-hour expiration timeline and automatically enforce tenant isolation by pre-binding the new employee to the inviting company's UUID."
* **Follow-up Questions:** *How do you prevent brute-force token attacks?* Answer: "By using cryptographically secure UUIDv4 tokens (128-bit space) which are virtually impossible to guess, and applying rate limits on the public check endpoint."

### 4. Policy CRUD with Flyway Versioned Migrations
* **What it does:** Allows HR to write, update, and manage company policies, ensuring that schema additions or modifications are tracked cleanly across development and production environments.
* **How it works technically:** We disabled Hibernate's `ddl-auto` (`ddl-auto: validate` in production). We write Flyway SQL migration scripts under `src/main/resources/db/migration` (e.g. `V1__init.sql`, `V4__add_company_invites.sql`). On application startup, Flyway automatically checks the `schema_version` metadata table in PostgreSQL and applies any pending migrations.
* **In an Interview:** Say: "We use Flyway to maintain database version control. This ensures database schema changes are treated exactly like code changes—fully versioned, tested, and automatically applied during deployment, preventing database sync failures."
* **Follow-up Questions:** *What happens if a migration script fails midway?* Answer: "Flyway rolls back the transaction (if the underlying database supports transactional DDL like PostgreSQL) and marks the migration as failed in the schema version table, requiring manual correction."

### 5. Consent Tracking and Audit Trail
* **What it does:** Keeps a compliance record showing which policies employees have accepted, and maintains an immutable audit log of actions.
* **How it works technically:** The `Consent` entity logs the `userId`, `policyId`, and `acceptedAt` timestamp. Whenever a policy is created, a task is completed, or a policy is accepted, the system records it in the `AuditLog` database table. The audit log queries are implicitly filtered by `company_id` via our Hibernate filter.
* **In an Interview:** Say: "To satisfy SOC2 and compliance audits, we implemented a digital signature signature-acceptance tracking system linked to an immutable, tenant-isolated audit logging database engine."
* **Follow-up Questions:** *Why is the audit log immutable?* Answer: "The backend does not expose any PUT or DELETE endpoints for the `AuditLog` entity, preventing logs from being tampered with by any tenant."

### 6. Onboarding Task Management and Document Upload
* **What it does:** Allows HR to assign tasks (like submitting I-9 forms) to employees, and lets employees upload documents (PDFs, images) to verify compliance.
* **How it works technically:** Documents are sent via multipart/form-data. The backend saves the files to a secure directory outside the web root (`platform/uploads/`), associates the file metadata path with the `OnboardingTask` or user profile in PostgreSQL, and enforces access control so users can only view their own uploaded files.
* **In an Interview:** Say: "Our task dashboard manages onboarding compliance. Users upload verification documents, which are stored securely on the filesystem while their metadata and paths are stored in PostgreSQL under tenant boundaries."
* **Follow-up Questions:** *How would you secure the file system storage against malicious file uploads?* Answer: "By verifying the file headers/magic numbers (not just the extension), restricting file sizes, renaming the files to random UUIDs upon saving to avoid directory traversal, and storing them outside the public web root."

### 7. RAG-Based AI Policy Assistant
* **What it does:** Allows employees to query an AI chatbot about company policies and get accurate answers derived directly from their company's documents.
* **How it works technically:** 
  1. **Embedding (Ingestion):** When an admin creates/updates a policy, they click "Embed". The backend splits the policy content into 500-character chunks. It calls the HuggingFace Inference API to get a 384-dimensional vector embedding for each chunk and saves them in `policy_chunks` table.
  2. **Retrieval:** When an employee asks a question, the backend embeds the question. It retrieves all policy chunks for that tenant, calculates the Cosine Similarity between the question vector and the chunk vectors in-memory, and picks the top 3 chunks.
  3. **Generation:** It constructs a system prompt enclosing these chunks as context and passes it to the Groq API (LLaMA-3.3-70b-versatile) to generate the final response.
* **In an Interview:** Say: "Instead of fine-tuning an expensive model, we built a Retrieval-Augmented Generation (RAG) assistant. It embeds policy text into vectors, performs real-time semantic similarity searches in PostgreSQL, and injects matching contexts into LLaMA via Groq."
* **Follow-up Questions:** *Why compute cosine similarity in Java instead of using pgvector?* Answer: "For initial project stages, in-memory computations on tenant-scoped chunks (which average a few dozen records per tenant) are extremely fast, cost-free, and remove heavy extension dependencies in PostgreSQL. As the database grows, we can easily swap to pgvector."

### 8. AI Policy Summarizer
* **What it does:** Auto-generates a concise 3-bullet-point summary of long policy documents.
* **How it works technically:** `PolicyService` sends the policy content to Groq with a system prompt restricting output to exactly 3 bullet points of up to 20 words each. The resulting summary is cached in Redis at key `summary:{policyId}` with a 60-minute TTL.
* **In an Interview:** Say: "To make compliance documents readable, we created an AI Summarizer. To optimize performance and cut API costs, we cache these generated summaries in Redis for 60 minutes."
* **Follow-up Questions:** *What happens when a policy is updated?* Answer: "When a policy is modified, the cache key `summary:{policyId}` is explicitly evicted from Redis so the next summary request triggers a fresh analysis."

### 9. AI Compliance Report Generator
* **What it does:** Allows Admins/HR to generate a summary report of all company compliance actions.
* **How it works technically:** Fetches all tenant audit logs, formats them into a text list, and sends them to Groq to write a professional compliance status summary. The report is cached in Redis under `report:{companyId}` for 30 minutes.
* **In an Interview:** Say: "Admins can generate audit summaries using Groq. The report covers gaps, risks, and audit progress, and is cached in Redis to minimize LLM usage overhead."

### 10. Conversational Memory in AI Chat
* **What it does:** Allows employees to have multi-turn conversations with the bot (e.g. asking follow-up questions).
* **How it works technically:** The React frontend maintains a state array of `messages` containing `{ role: 'user' | 'assistant', content: String }`. On each new query, it sends the full array as `history` in the request body to `POST /api/ai/ask`. `RagService` bypasses RAG single-question caching when history is present, constructs the full dialogue messages array, and sends it to Groq.
* **In an Interview:** Say: "We implemented conversational memory by holding context on the client and transferring history arrays to the API, allowing our RAG assistant to maintain contextual awareness across chat sessions."

### 11. Compliance Risk Score with AI Analysis
* **What it does:** Evaluates company risk based on policy acceptance rates, mapping a risk score and level (LOW/MEDIUM/HIGH) alongside AI recommendations.
* **How it works technically:** Implemented at `GET /api/compliance/risk-score`. It calculates acceptance rates per policy (Consents / Total Employees * 100), feeds this data to Groq to generate recommendations, and computes a master score: `(total consents / (total policies * total employees)) * 100`. It caches the JSON response in Redis at key `risk-score:{companyId}` for 30 minutes, allowing bypass via a `?refresh=true` query parameter.
* **In an Interview:** Say: "We calculate mathematical compliance scores based on policy consents and run LLaMA risk analysis over the data to suggest corrections. The score is cached client-side and in Redis, with a force-refresh option."

### 12. Redis Caching
* **What it does:** Speeds up response times, acts as a security blacklist registry, and reduces LLM API token consumption.
* **How it works technically:** Handles:
  1. Revoked JWTs (expiration equal to remaining token time).
  2. Single RAG answers (key `rag:{companyId}:{question}` for 15 mins).
  3. Policy summaries (key `summary:{policyId}` for 60 mins).
  4. Compliance reports (key `report:{companyId}` for 30 mins).
  5. Compliance risk scores (key `risk-score:{companyId}` for 30 mins).
* **In an Interview:** Say: "Redis is our performance buffer. It handles JWT blacklists for fast auth checks and caches Groq responses to shield us from external API rate limits and costs."

### 13. Role-Based Access Control (RBAC) with `@PreAuthorize`
* **What it does:** Prevents unauthorized users from accessing REST endpoints.
* **How it works technically:** `JwtAuthFilter` extracts the role from the JWT, prefixes it with `ROLE_`, and inserts it into Spring's security context. We use Spring Security's method security, marking controller methods with annotations like `@PreAuthorize("hasAnyRole('ADMIN', 'HR')")`.
* **In an Interview:** Say: "We secure our API endpoints at the method level using Spring Security's `@PreAuthorize` expression engine, mapping user permissions directly to JWT authority claims."

### 14. React TypeScript Frontend with Role-Specific Dashboards
* **What it does:** Provides a highly premium, type-safe user interface that adapts dynamically to user roles.
* **How it works technically:** Built using Vite, TypeScript, and React Router. An `AuthContext` holds the authenticated user payload. Routes are protected via route guards. Admins see the compliance audit/onboarding panels; employees see tasks, consents, and RAG chats.

## 3. Architecture Deep Dive

### Why Shared-Schema Multi-Tenancy over Schema-per-Tenant?
1. **Cost & Resource Efficiency:** On Render Free Tier or cloud hosting, running separate database schemas or database instances for hundreds of tenants is cost-prohibitive. A shared-schema design uses a single connection pool and database instance.
2. **Simplified Migrations:** Schema changes are executed once via Flyway, rather than needing to run migrations across hundreds of separate schemas.

### Why UUID Primary Keys over Auto-Increment Integers?
1. **Security (Idempotency and Enumeration Prevention):** Auto-incrementing IDs (e.g. `policies/1`, `policies/2`) allow malicious users to guess other companies' resource URLs. UUIDv4 keys (128-bit) are non-sequential and impossible to guess.
2. **Easy Multi-Tenant Data Merging:** If we ever need to migrate or merge database tables, UUIDs prevent key collisions.

### Why Flyway over Hibernate `ddl-auto`?
1. **Production Safety:** Letting Hibernate auto-generate schemas (`ddl-auto: update` or `create`) in production can lead to accidental column drops, data loss, or locked tables.
2. **Audit Trails:** Flyway migrations are explicit, SQL-based scripts committed to git, leaving a clear version history of database evolutions.

### Why Redis for JWT Blacklisting instead of Database?
1. **Performance:** A relational DB query on every API call to check blacklisted tokens causes heavy disk I/O. Redis performs entirely in-memory and handles high read throughput with sub-millisecond latencies.
2. **Automatic Cleanups:** Redis `EXPIRE` automatically deletes blacklisted tokens once the token's original TTL expires, preventing database storage bloating.

### Why Cosine Similarity in Java instead of pgvector?
1. **Infrastructure Simplicity:** Using the `pgvector` extension requires custom database setups and compiles. For small to mid-sized policy repositories (each company typically has under 50 policies), in-memory Java math on a pre-filtered tenant list is extremely fast, free, and lightweight.
2. **Portability:** The application can run on any standard PostgreSQL instance without requiring custom extensions.

### Why Groq over OpenAI for LLM Inference?
1. **Unmatched Speed:** Groq's LPU (Language Processing Unit) architecture serves LLaMA-3.3 completions at over 500 tokens per second, giving our UI an instant, near-zero-latency feel.
2. **Cost-Efficiency:** Groq offers generous free-tier quotas for development, reducing startup costs compared to OpenAI's paid API keys.

### Why RAG over Fine-Tuning for the AI Assistant?
1. **Hallucination Control:** Fine-tuned models still hallucinate. RAG provides the exact policy text chunk to the model as context, ensuring answers are fact-based and verified.
2. **Dynamic Updates:** If a policy changes, we simply re-embed that policy. Fine-tuning would require retraining the entire model, which is expensive and time-consuming.
3. **Tenant Privacy:** With RAG, we only retrieve policy chunks belonging to the current tenant's `company_id`. Fine-tuning a single model on all companies' data risks cross-tenant data leakage.

## 4. Technology Stack Explanation

* **Spring Boot 3.5:** Core Java framework. Chosen for its production-grade MVC container, built-in dependency injection, and native integration with JPA/Hibernate.
* **Spring Security:** Enterprise security framework. Handles token parsing, cors configurations, stateless sessions, and role-based endpoint routing.
* **JWT (jjwt-api):** JSON Web Token library. Used to issue and parse stateless tokens carrying user identities and tenant IDs securely.
* **Hibernate/JPA:** Object-Relational Mapper. Maps Java classes to PostgreSQL tables, handles connections, and runs SQL modifications.
* **PostgreSQL:** ACID-compliant relational database. Sturdily handles transactions, relationships, and structured policy/user tables.
* **Flyway:** Schema migration tool. Automatically applies versioned SQL patches to PostgreSQL on startup.
* **Redis:** In-memory data store. Used for session revocation lists and response caching to keep latency low.
* **WebClient (Spring WebFlux):** Reactive HTTP client. Used to make asynchronous non-blocking calls to the HuggingFace and Groq APIs.
* **React & TypeScript:** Single-page frontend framework. TypeScript ensures type-safety, preventing null-pointer errors in the UI, and React provides dynamic components.
* **Vite:** Next-generation frontend build tool. Replaces slow Webpack builds, compile times are under 300ms, and it builds production bundles in seconds.
* **Axios:** HTTP client for the browser. Handles API call interceptors to automatically append JWT Bearer headers to outgoing requests.

## 5. Sure-Shot Interview Questions with Answers

### Technical Questions

#### 1. What is multi-tenancy and how did you implement it?
"Multi-tenancy is an architecture where a single instance of software serves multiple customers (tenants). We implemented a shared-database, shared-schema tenant model. Every tenant-scoped database table includes a `company_id` UUID column. When a user logs in, their JWT encodes their `companyId`. We extract this claim in `JwtAuthFilter` and bind it to a thread-safe `TenantContext` using `ThreadLocal`. Finally, we wrote a `TenantFilterAspect` that intercepts repository calls, unwraps the Hibernate session, and automatically applies the `@FilterDef("tenantFilter")` with the current `company_id` to ensure SQL queries are isolated by default."

#### 2. Explain how JWT authentication works in your project.
"When a user signs in, their credentials (email, password, company name) are verified by `AuthService`. If correct, the backend generates a JWT signed with a secret key using HMAC SHA-256. The token contains claims for `userId`, `companyId`, and `role`. On subsequent API requests, the React client attaches this token in the `Authorization: Bearer <token>` header. Our `JwtAuthFilter` intercepts the request, checks that the token is valid and not expired, verifies it isn't listed in our Redis logout blacklist, and populates Spring's `SecurityContext` with an `AuthenticatedUser` principal."

#### 3. What is RAG and how does your AI assistant use it?
"RAG stands for Retrieval-Augmented Generation. When an Admin embeds a policy, the backend breaks it into chunks, generates vector embeddings using a HuggingFace model, and stores them in PostgreSQL. When an employee asks a question, the system embeds the question, fetches all policy chunks for that tenant, computes the Cosine Similarity between vectors in-memory, and retrieves the top 3 chunks. It then injects this text context into our prompt to the Groq LLaMA model, forcing the LLM to answer using only the retrieved policies."

#### 4. How does Redis caching improve your application?
"Redis acts as our primary caching layer. We use it to store revoked JWTs on logout so we don't hit PostgreSQL for auth checks. We also cache RAG completions, policy summaries, and compliance reports. Since LLM queries take several seconds and incur API tokens, caching these results in Redis (with TTLs of 15 to 60 minutes) yields instantaneous responses for duplicate requests and reduces external API costs."

#### 5. What is the Hibernate tenant filter and how does it prevent data leakage?
"It is a Hibernate annotation that allows us to define parameterized filters at the entity class level. By declaring `@FilterDef(name = "tenantFilter")` on our `BaseEntity` mapped superclass, Hibernate automatically appends `AND company_id = :companyId` to all generated SELECT queries. This prevents developers from forgetting to restrict database queries by tenant, securing the platform against cross-tenant data leakage."

#### 6. Explain the difference between authentication and authorization in your project.
"Authentication is verifying *who* the user is, handled by `AuthController` and verified by checking the JWT signature in `JwtAuthFilter`. Authorization is verifying *what* the user is allowed to do. We enforce authorization using Spring Security roles (`ADMIN`, `HR`, `EMPLOYEE`). Methods in controllers are secured with `@PreAuthorize` annotations, ensuring, for example, that only Admins and HR can access `/api/audit/report` or `/api/compliance/risk-score`."

#### 7. How does cosine similarity work for finding relevant policy chunks?
"Cosine similarity measures the cosine of the angle between two multi-dimensional vectors, resulting in a value between -1 and 1. In our `RagService`, the vectors are 384-dimensional arrays representing the semantic meaning of text. We calculate the dot product of the question vector and chunk vector, divided by the product of their magnitudes. Chunks with scores closest to 1 represent the closest semantic context matches to the user's question."

#### 8. What happens when a user logs out in your system?
"Since JWTs are stateless, they cannot be deleted from the server. To handle logout, the client calls `/api/auth/logout`. The backend reads the token, calculates its remaining validity duration (expiration time minus current time), and inserts the token as a key into Redis with that duration as the TTL. During any request, `JwtAuthFilter` checks Redis. If the key exists, the request is rejected with `401 Unauthorized`, effectively revoking the token."

#### 9. Why did you use Flyway instead of letting Hibernate manage the schema?
"Using Hibernate's auto-ddl in production is a major risk because it can execute destructive DDL queries (like dropping tables) automatically. Flyway provides absolute control. Database migrations are written as explicit SQL scripts, committed to version control, and applied sequentially. This makes database schema modifications reproducible, testable, and safe across different staging environments."

#### 10. How does the invite-based registration maintain multi-tenant isolation?
"When an HR user generates an invite link, the backend creates a `CompanyInvite` containing a secure token and binds it to the HR user's `companyId`. When the employee clicks the link and submits their registration, the backend matches the token, reads the pre-bound `companyId` from the invite record, and writes the new employee with that exact `companyId`. The registering employee cannot manipulate their company association, securing tenant isolation."

### System Design Questions

#### 11. How would you scale this system to 1000 companies?
1. **Database Partitioning:** Implement PostgreSQL table partitioning by list using the `company_id` column. Queries will automatically route to the correct partition, keeping indexes small and fast.
2. **Database Replication:** Set up PostgreSQL read replicas. Since HR platforms are read-heavy (employees reading policies and checking tasks), we can route all SELECT queries to replicas and write queries to the primary database.
3. **Dedicated Vector Search:** Swap out the in-memory Java cosine similarity with a dedicated vector database or extension like `pgvector` or Qdrant to handle million-scale vector matching.
4. **Caching Layer:** Scale Redis into a cluster configuration to support high concurrency.

#### 12. How would you handle if the Groq API is down?
1. **Fallback LLM Provider:** Implement a fallback service in the code (e.g. WebClient switching to HuggingFace, OpenAI, or a local Ollama instance if Groq returns 5xx errors).
2. **Graceful UI Degradation:** Catch API errors and return a user-friendly message like *"Our AI assistant is undergoing brief maintenance, showing cached answers only."*
3. **Retry Mechanisms:** Wrap external WebClient calls in a Spring Retry/Resilience4j circuit breaker with exponential backoff.

#### 13. How would you add email notifications to this system?
1. **Event-Driven Architecture:** Publish events (like `InviteGeneratedEvent` or `TaskAssignedEvent`) to an asynchronous event bus (such as Spring Events, RabbitMQ, or Apache Kafka).
2. **Asynchronous Mail Worker:** Create a listener class that consumes these events, constructs email templates using Thymeleaf or FreeMarker, and dispatches them asynchronously using Spring's `JavaMailSender` or an external service like SendGrid, ensuring the main HTTP request thread remains unblocked.

#### 14. How would you implement rate limiting on the AI endpoints?
1. **API Gateway Rate Limiting:** Implement rate limiting at the gateway level (e.g. Spring Cloud Gateway or Nginx).
2. **Redis Token Bucket Rate Limiter:** Use Redis with a Lua script (or Spring Boot's Bucket4j library backed by Redis) to implement a token bucket rate limiter per user/IP, allowing us to limit requests to, say, 5 queries per minute for the AI chat.

#### 15. How would you add a mobile app to this system?
"Since the backend is a stateless REST API secured by JWT, it is completely decoupled from the frontend. We can build a React Native, Flutter, or native Android/iOS app. The mobile client would simply authenticate against our `/api/auth/login` endpoint, store the JWT in secure storage (like Keychain or Shared Preferences), and call the exact same REST endpoints as our web client."

"The most complex issue was a `ClassNotFoundException: AuthenticatedUser` during local restarts. Spring Boot DevTools reloads application classes using a custom `RestartClassLoader` to speed up changes. However, Tomcat's session manager was persisting sessions containing the security context principal to disk on shutdown, and attempting to deserialize it using the base classloader on startup, resulting in classloader mismatch errors. I fixed this by making `AuthenticatedUser` implement `Serializable` with a stable `serialVersionUID`, and disabling Tomcat's session persistence in `application.yml` by setting `server.servlet.session.persistent` to `false`."

## 6. Key Technical Terms Glossary

* **Multi-Tenancy:** A software architecture where a single application serving multiple customer companies divides data views so that each company only interacts with their own records.
* **Tenant Isolation:** Enforcing boundaries so that data belonging to Company A is completely invisible to users of Company B.
* **Hibernate Filter:** A feature that appends SQL constraints (like `WHERE company_id = ?`) to all database queries automatically.
* **JWT (JSON Web Token):** A stateless token containing JSON claims cryptographically signed to securely authenticate requests.
* **Redis:** An in-memory key-value cache database used for quick checks and revoking credentials.
* **RAG (Retrieval-Augmented Generation):** An AI framework where an LLM is provided with relevant documents as context to answer questions.
* **Embeddings:** Vector representations of words or sentences in numerical space, mapping semantic meanings.
* **Cosine Similarity:** A metric calculation used to find how semantically close two text vectors are.
* **Vector Search:** Querying database vectors to find the most relevant contextual text.
* **LLM (Large Language Model):** A deep learning model trained on text data to understand and generate human-like text.
* **RBAC (Role-Based Access Control):** Restricting systems access to authorized users based on their assigned roles.
* **Spring Security:** A powerful framework providing authentication, authorization, and protection against common attacks.
* **Flyway Migration:** Incremental SQL scripts executing database modifications in sequence.
* **Connection Pooling:** Keeping a pool of database connections open to reuse, avoiding connection overhead.
* **Stateless Authentication:** Validating client requests by verifying JWT claims on every request, requiring no server session state.
* **Token Blacklisting:** A registry storing revoked tokens until their natural expiration to block logged-out sessions.
* **CORS (Cross-Origin Resource Sharing):** A security mechanism restricting web browsers from making requests to a different domain.
* **BCrypt:** A secure password-hashing function using salt parameters to prevent rainbow table attacks.
* **WebClient:** A modern, reactive HTTP client used to fetch remote APIs in Spring Boot.
* **DTO (Data Transfer Object) Pattern:** A design pattern encapsulating data passed between REST layers, hiding internal database models.

## 8. What Makes This Project Stand Out

1. **AOP Tenant Scoping:** Enabling Hibernate filters dynamically via an Aspect-oriented Aspect rather than appending SQL conditions in services prevents human error and data leakage.
2. **Redis JWT Blacklist:** Implementing a high-performance revoked token registry in Redis solves the security problem of stateless JWT logouts without incurring SQL performance overhead.
3. **Flyway versioning:** MCA graduates often use `ddl-auto: update`. Using version-controlled database migrations shows corporate and production readiness.
4. **Hybrid Cosine Similarity:** Evaluating similarity in Java on pre-filtered tenant lists shows analytical thinking—balancing infrastructure complexity against current project scale.
5. **Caching LLM Results:** Caching Groq AI completions in Redis with varying TTL rules shows deep consideration for API rate boundaries and cost management.

## 10. Common Mistakes to Avoid in Interview

1. **"I let Hibernate generate the tables."**
   * *Instead say:* "We disabled Hibernate `ddl-auto` in favor of Flyway migrations, ensuring DDL changes are version-controlled and reproducible."
2. **"I used sessions to store logged-in users."**
   * *Instead say:* "Our backend is stateless. We authenticate requests using cryptographically verified JWT tokens passed in authorization headers."
3. **"To query tenant data, I manually added `WHERE company_id = ...` to every query."**
   * *Instead say:* "We decoupled multi-tenancy logic from business code using Hibernate filters and an AOP aspect, making tenant isolation completely automatic and transparent."
4. **"I fine-tuned a model to learn company policies."**
   * *Instead say:* "We built a Retrieval-Augmented Generation (RAG) architecture because fine-tuning is expensive, prone to hallucinating facts, and risks leaking confidential tenant data."
5. **"Redis is just a database."**
   * *Instead say:* "Redis is an in-memory key-value cache that we use to cache high-cost AI responses and store revoked JWT tokens with auto-expiring TTLs."
6. **"I used auto-increment IDs because they are simple."**
   * *Instead say:* "We used UUIDv4 primary keys to prevent ID enumeration attacks and ensure unique keys across tenant environments."
7. **"I didn't care about security in local development."**
   * *Instead say:* "We stored API tokens and database keys in environment variables, separating configurations from the main codebase."
8. **"A constructor is just for setting values."**
   * *Instead say:* "We used Lombok's `@RequiredArgsConstructor` to implement constructor-based dependency injection, which is Spring's recommended practice for easier unit testing."
9. **"I created one big page with all code."**
   * *Instead say:* "We separated our React components into pages, layout wrappers, context providers, and service modules, keeping the frontend scalable and modular."
10. **"The backend and frontend are tightly coupled."**
    * *Instead say:* "The frontend and backend are completely decoupled. They communicate exclusively through stateless JSON REST APIs, meaning the backend could easily support a mobile client."
